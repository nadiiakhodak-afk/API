PaySaxas API uses cursor-based pagination to efficiently handle large datasets and to ensure consistent and predictable responses when listing
resources.
In addition to pagination, supported endpoints also provide a set of query filters that allow clients to narrow the results by account, date, direction, or
status

## Common Query Parameters
| Parameter | Type    | Applies to | Description / Example                                                               |
|-----------|---------|------------|-------------------------------------------------------------------------------------|
| `limit`   | integer | all        | Maximum number of items per page (`default: 20`, `max: 100`). Example: `?limit=50`. |
| `cursor`  | string  | all        | Encoded pointer to the next result set. Example: `?cursor=eyJpZCI6IjEyMyJ9`.        |

## Query Parameters (Filtering)

All pagination endpoints may support filtering.
Filters allow clients to request only the relevant subset of data.

| Parameter  | Type                | Required | Description                                                                           |
|------------|---------------------|----------|---------------------------------------------------------------------------------------|
| account_id | string              | yes      | Return transactions only for a specific customer account.                             |
| date_from  | string (YYYY-MM-DD) | no       | Return records created on or after this date (UTC)                                    |
| date_to    | string (YYYY-MM-DD) | no       | Return records created on or before this date (UTC).                                  |
| status     | string              | no       | Filter by unified lifecycle status: pending, processing, completed, failed, returned. |
| type       | string              | no       | Transaction direction: incoming or outgoing.                                          |
| order      | string              | no       | Result ordering. Only desc is supported (default).                                    |
| id         | string              | no       | Return specific record(s) by identifier (optional helper filter).                     |

### Notes
- Date filters must be provided without time. Format: YYYY-MM-DD.
- All filtering is performed before pagination.
- Filters that are not provided are silently ignored (empty values allowed; must not return null)

### Example request
```
GET /v1/transactions?account_id=acc_32c410&date_from=2025-01-01&limit=30
```

>Real account or IBAN values are never exposed in documentation, examples, or logs.
>Only structural patterns are demonstrated


## Pagination metadata
| Field                      | Type    | Description                                                                    |
|----------------------------|---------|--------------------------------------------------------------------------------|
| next_cursor                | string  | Encoded pointer to request the next page. Example: ```?cursor=<next_cursor>``` |
| previous_cursor (optional) | string  | Encoded pointer to request the previous page (if supported).                   |
| has_more (optional)        | boolean | Indicates if additional pages exist.                                           |
| page_size (optional)       | integer | Number of items returned in this response.                                     |


## Best Practices
- Always use a cursor and a limit for paginated requests.
- Do not use offset at any endpoint.
- Validate that the limit stays within the allowed range (default: 20, max: 100).
- Always check next_cursor before fetching the next page.
- Use UTC timestamps in ISO 8601 format.
- Avoid exposing cursor tokens in URLs, logs, or analytics.
- Prefer pagination or asynchronous exports when downloading large datasets.
